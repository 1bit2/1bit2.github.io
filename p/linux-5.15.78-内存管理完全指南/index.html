零、为什么需要虚拟内存

在理解内核实现之前，先明确虚拟内存解决的根本问题：

物理内存的困境：如果进程直接使用物理地址，那么——

  1. 进程 A 可以通过指针访问进程 B 的数据（无隔离）
  2. 内存碎片化后无法分配连续大块（无连续视图）
  3. 每个进程必须知道自己代码加载在哪个物理地址（无法重定位）

虚拟内存的解法：给每个进程一个独立的、从零开始的地址空间。内核通过页表将虚拟地址映射到物理地址，实现：

需求实现方式
进程隔离每个进程有自己的页表，看不到其他进程的物理页
连续视图虚拟地址连续，物理页可以分散在任意位置
按需分配VMA 只记账虚拟范围，首次访问时才通过缺页异常分配物理页
共享多个进程的页表项指向同一物理页（如 libc.so 的代码段）

核心机制链：VMA（§二）→ mmap/brk（§五）→ 缺页异常（§九）→ 物理页分配（§八）→ 回收/OOM（§十）。其中缺页异常是串联一切的枢纽——它把虚拟地址空间的记账、页表翻译、物理页分配全部连接起来。

为什么大页能提高性能？

CPU 内的 TLB 缓存虚拟→物理地址的翻译结果，但容量很小（典型 x86 CPU 约 64 条数据 TLB 表项）：

页大小覆盖 8MB 内存需要 TLB 表项miss 时查页表级数
4KB 普通页2048 页2048 条 → 远超容量 → 频繁 miss4 级
2MB 大页4 页4 条 → 完全够用 → 高命中3 级（省 PTE）
1GB 大页1 页1 条2 级（省 PMD+PTE）

页越大 → 覆盖同样内存的 TLB 条目越少 → 命中率越高 → 地址翻译越快。

14.3 建议的学习阶段

阶段1 (1~2周): 建立全局视图

  • 读 mm_types.h 的 mm_struct 和 vm_area_struct
  • 用 /proc/<pid>/maps 观察真实进程 VMA 布局
  • 读 mm/mmap.c 的 do_mmap → mmap_region 流程

阶段2 (2~3周): 理解缺页异常

  • 读 arch/x86/mm/fault.c → mm/memory.c 的完整链路
  • 分别理解 do_anonymous_page/do_fault/do_wp_page/do_swap_page

阶段3 (2~3周): 理解物理内存管理

  • 伙伴系统: __alloc_pages → get_page_from_freelist → __rmqueue_smallest
  • SLUB: kmem_cache_alloc → slab_alloc_node → ___slab_alloc

阶段4 (2~3周): 理解回收与OOM

  • kswapd → balance_pgdat → shrink_node → shrink_lruvec
  • out_of_memory → oom_badness → oom_kill_process

阶段5 (持续): 进阶专题

  • 透明大页(THP)、内存cgroup、NUMA、KSM、内存压缩
