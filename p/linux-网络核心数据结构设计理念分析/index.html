Linux 网络核心数据结构设计理念分析

本文分析内核中六个核心网络结构的设计思想：sk_buff、net_device、sock、inet_sock、inet_connection_sock、tcp_sock。

一、struct sk_buff —— “包”的载体

1.1 定位

  • 角色：表示一个网络数据包在协议栈中的完整生命周期。
  • 设计目标：在各层协议之间传递时，既能携带数据，又能携带每层需要的元数据，且避免频繁拷贝。

1.2 核心设计理念

理念实现方式
统一包描述一个结构同时描述 L2/L3/L4 头位置、数据区、校验和状态、设备、socket 等，各层通过 skb_push/skb_pull 和 *_header 偏移访问自己的头。
零拷贝/少拷贝线性区 + frags[] + frag_list 的 scatter-gather 布局；head/data/tail/end 只在指针上移动，不搬数据。
每层私有元数据48 字节 cb[] 控制缓冲区，各层用宏（如 TCP_SKB_CB、IPCB）映射到自己的控制块，避免为每层单独分配。
多用途链表节点开头的 next/prev 与 sk_buff_head 兼容，同时用 union 支持 rb_node、list_head，同一结构可挂到发送队列、重传队列、GRO 等不同容器。
引用与克隆users 引用计数；cloned/fclone 区分共享数据区的克隆，释放时通过 destructor 回调和 sk_*_alloc 做 socket 内存计费。

1.4 小结

sk_buff 的设计是**“一个包、一块元数据、多级缓冲区”**：用一份描述符贯穿驱动→L2→L3→L4→socket，通过指针与偏移而非拷贝在各层间传递，并兼顾硬件卸载（校验和、GSO、XDP 等）和统计、安全、TC 等扩展需求。

二、struct net_device —— “设备”的抽象

2.1 定位

  • 角色：表示一个网络接口（物理或虚拟），是内核与驱动、协议栈与硬件的桥梁。
  • 设计目标：对上层提供统一的“网卡”抽象，对驱动提供明确的操作接口和配置/状态存放处。

2.2 核心设计理念

理念实现方式
统一设备抽象所有设备（以太网、VLAN、bond、veth、tun 等）都是 net_device，通过 netdev_ops、ethtool_ops、header_ops 等函数表区分行为。
协议无关 + 协议扩展核心字段（MTU、flags、队列、状态）与协议无关；协议相关通过 ip_ptr、ip6_ptr、ieee80211_ptr 等指针挂到设备上，便于协议栈按需访问。
快速路径与缓存行注释中强调 “Read-mostly cache-line for fast-path”：flags、netdev_ops、ifindex、mtu、hard_header_len 等收发包路径常用字段集中，减少 cache miss。
多队列与 RSS/RPSnum_rx_queues、real_num_rx_queues、_rx、发送队列等支持多队列网卡和 RSS/RPS 流分发；queue_mapping 在 skb 上记录队列，便于亲和性。
可配置与可观测features/hw_features/wanted_features 表示能力与配置；统计与丢弃计数（如 rx_dropped、tx_dropped）便于监控和排错。

2.3 与协议栈的边界

  • 注释中的 FIXME：“cleanup struct net_device such that network protocol info moves out” 表明长期方向是设备与协议解耦：设备管队列、DMA、中断、特性；协议相关状态放在协议自己的结构里，通过指针关联。
  • 当前做法：设备仍持有 ip_ptr、ip6_ptr 等，协议栈在收包、路由、邻居等路径上通过 dev 找到这些指针，属于历史与性能的折中。

2.4 小结

net_device 的设计是**“一个设备、一套操作表、多协议扩展”**：用统一结构描述所有网卡，用函数表和协议指针实现差异化和扩展，并照顾快速路径的缓存布局与多队列、可观测性。

三、struct sock —— “连接/端点”的通用抽象

3.1 定位

  • 角色：表示一个通信端点（socket 在内核侧的对应物），既可用于面向连接（TCP），也可用于无连接（UDP/RAW）。
  • 设计目标：提供协议族无关的公共部分（状态、队列、内存、回调），让各协议族（INET、INET6、Unix 等）在此基础上扩展。

3.2 核心设计理念

理念实现方式
公共头 + 协议扩展第一个成员是 struct sock_common __sk_common，包含地址对、端口对、哈希、状态、引用计数、协议操作表等；sock 在此基础上增加队列、定时器、回调、缓冲区限制等，供所有协议共用。
与 timewait 共享布局注释明确：“inet_timewait_sock 也使用 sock_common，因此不要在 __sk_common 之前添加任何字段”，保证 sock 与 inet_timewait_sock 在公共部分布局一致，便于查找和转换。
双队列 + 背压sk_receive_queue：已到达未读数据；sk_write_queue：待发送数据；sk_backlog：当 socket 被用户锁定时软中断暂存包，避免丢包；sk_rmem_alloc/sk_wmem_alloc 与 skb 的 destructor 联动，实现背压与计费。
回调驱动的事件模型sk_data_ready、sk_write_space、sk_state_change、sk_error_report、sk_backlog_rcv、sk_destruct 等由协议或上层设置，在数据到达、可写、状态变化、错误、背压处理、释放时统一回调，解耦协议与唤醒逻辑。
路由与特性缓存sk_dst_cache、sk_rx_dst 缓存路由，避免每包查表；sk_route_caps/sk_gso_max_size 等缓存设备能力，用于 GSO/校验和等决策。

3.3 sock_common 的复用

  • sock_common 设计为“最小网络层 socket 头”：地址对、端口对、哈希、状态、绑定节点、协议指针、网络命名空间、引用计数等。
  • 被 sock、inet_timewait_sock、request_sock 等复用，通过 union 在不同阶段复用同一偏移的字段（如 skc_listener、skc_tw_dr、skc_rcv_wnd），在保持布局稳定的前提下节省内存并统一查找路径。

3.4 小结

sock 的设计是**“公共端点 + 双队列 + 回调 + 布局兼容”**：用 sock_common 做最小公共头并与 timewait 等共享布局，用 sock 承载队列、内存计费、路由缓存和事件回调，使各协议族只实现自己的扩展部分。

四、struct inet_sock —— “IP 端点”的抽象

4.1 定位

  • 角色：在 sock 之上增加 IPv4（及双栈） 所需的地址、选项、多播、cork 等，是 INET 协议族中 TCP 与 UDP 的公共父类。
  • 设计目标：把与“IP 地址、端口、TOS、多播、IP 选项”相关的共性放在一层，避免在 TCP/UDP 中重复实现。

4.2 核心设计理念

理念实现方式
继承关系明确第一个成员是 struct sock sk，注释要求“必须是第一个成员”，这样 struct inet_sock * 可安全转为 struct sock *，符合 C 的“结构体嵌入即继承”的用法。
地址端口统一视图通过宏把 sk.__sk_common 的 skc_daddr/skc_rcv_saddr/skc_dport/skc_num 暴露为 inet_daddr/inet_rcv_saddr/inet_dport/inet_num，再增加 inet_saddr/inet_sport 等，形成“发送用 saddr/sport，接收/查找用 rcv_saddr/num”的清晰语义。
IP 层选项与行为inet_opt（IP 选项）、tos、uc_ttl、min_ttl、pmtudisc、recverr、hdrincl、transparent 等对应 setsockopt/getsockopt 和收发包行为，只在这一层出现，TCP/UDP 共用。
多播与 corkmc_list、mc_ttl、mc_index、mc_addr 等支持多播；cork 用于 UDP/RAW 等“先积累再一次性发”的优化，避免每小包都走完整路由和分片路径。
双栈扩展点#if IS_ENABLED(CONFIG_IPV6) 下的 pinet6 指向 ipv6_pinfo，在不改变 inet_sock 主布局的前提下为 IPv6 双栈提供扩展。

4.3 小结

inet_sock 的设计是**“sock + IP 地址/选项/多播/cork”**：严格继承 sock，把 INET 共有概念集中在一层，为 TCP 和 UDP 提供统一基类，并为 IPv6 双栈留出指针扩展。

五、struct inet_connection_sock —— “面向连接”的抽象

5.1 定位

  • 角色：在 inet_sock 之上增加面向连接协议共有的概念：accept 队列、重传/延迟 ACK 定时器、RTO、拥塞控制接口、MTU 探测、ULP 等，主要为 TCP 服务，也兼顾 DCCP 等。
  • 设计目标：把“连接建立、重传、ACK、拥塞、MTU、上层协议”等与“连接”强相关的逻辑从具体协议中抽出来，放在一层可复用的结构里。

5.2 核心设计理念

理念实现方式
继承 inet_sock第一个成员是 struct inet_sock icsk_inet，“父类必须是第一个成员”，保证 inet_connection_sock * → inet_sock * → sock * 的向上转换有效。
连接建立与 accepticsk_accept_queue 是已完成握手的子连接队列；与 request_sock、syn_recv_sock 等配合，实现 listen/accept 与三次握手。
定时器与 RTOicsk_retransmit_timer、icsk_delack_timer、icsk_rto/icsk_rto_min、icsk_delack_max 等统一管理重传与延迟 ACK，避免每个连接型协议各自实现一套。
拥塞控制插件化icsk_ca_ops、icsk_ca_state、icsk_ca_priv[] 等把拥塞算法抽象成可插拔的 tcp_congestion_ops，便于 Cubic、Reno、BBR 等切换和扩展。
地址族与 ULP 扩展icsk_af_ops 封装与地址族相关的发送、连接请求、syn_recv_sock 等；icsk_ulp_ops/icsk_ulp_data 为 TLS 等上层协议提供挂载点。
延迟 ACK 与 MTU 探测icsk_ack 子结构集中管理 pending/quick/pingpong/ato 等；icsk_mtup 管理 PMTU 探测的 search_high/low、probe_size、enabled 等，与路径 MTU 发现和 MSS 同步配合。

5.3 小结

inet_connection_sock 的设计是**“inet_sock + 连接队列 + 定时器 + 拥塞/ULP/MTU 抽象”**：在 IP 端点之上专门为“有连接、有重传、有拥塞控制”的协议提供一层公共实现，使 TCP 只需关注序列号、窗口、SACK 等 TCP 特有状态。

六、struct tcp_sock —— “TCP 连接”的完整状态

6.1 定位

  • 角色：表示一条 TCP 连接的完整状态，包括 RFC 793 的序列号/窗口、拥塞控制、重传、RTT、选项、TSQ、Fast Open 等，是 TCP 实现的唯一“大状态机”载体。
  • 设计目标：在继承 inet_connection_sock 的基础上，把所有 TCP 专有 的字段集中在一个结构里，便于维护和与 RFC 对照。

6.2 核心设计理念

理念实现方式
继承 inet_connection_sock第一个成员是 struct inet_connection_sock inet_conn，保证 tcp_sock * → inet_connection_sock * → … → sock * 的转换有效，协议层和通用代码可以只拿 struct sock *，在需要时用 tcp_sk() 转为 tcp_sock *。
RFC 793 变量集中snd_una、snd_nxt、rcv_nxt、copied_seq、snd_wnd、rcv_wup、max_window、window_clamp 等与 RFC 命名一致，便于与标准对照；发送/接收窗口、已读/已确认位置一目了然。
发送/接收与统计bytes_sent/bytes_acked/bytes_received、segs_in/segs_out、data_segs_in/data_segs_out 等用于统计和调试；dsack_dups、reordering、reord_seen 等支持 SACK 与重排序检测。
RTT 与拥塞srtt_us、mdev_us、rttvar_us、rtt_min 等用于 RTO 计算；snd_cwnd、snd_ssthresh、packets_out、retrans_out 等与拥塞控制算法（由 icsk_ca_ops 提供）配合。
快速路径与预测pred_flags 用于“头部预测”：若收到的包符合预期（正常数据或 ACK），可跳过部分检查以加速常见路径。
扩展与特性rx_opt 存放接收到的 TCP 选项；tsq_node、tsorted_sent_queue 支持 TSQ 和按时间排序；save_syn、syn_data、syn_fastopen* 等支持 TFO 和诊断。

6.3 小结

tcp_sock 的设计是**“inet_connection_sock + 完整 TCP 状态”**：在连接层公共逻辑之上，把所有 TCP 专有的序列号、窗口、RTT、拥塞、选项、TSQ、TFO 等集中在一个结构里，通过“第一个成员是父类”的继承链与 sock 层统一，既满足协议正确性，又便于阅读 RFC 和扩展新特性。

7.1 设计脉络总结

  1. sk_buff：数据包的单一描述符，跨层传递，少拷贝、可扩展（cb、extensions）、可挂队列/树。
  2. net_device：设备的单一抽象，操作表 + 协议指针，兼顾快速路径与多队列/可观测。
  3. sock 系：端点/连接的层次化抽象：
    • sock_common：最小公共头，与 timewait/request 等共享布局。
    • sock：通用端点（队列、内存、回调、路由缓存）。
    • inet_sock：+ IP 地址、端口、选项、多播、cork。
    • inet_connection_sock：+ 连接队列、重传/延迟 ACK、拥塞接口、ULP、MTU 探测。
    • tcp_sock：+ 完整 TCP 状态（序列号、窗口、RTT、SACK、TSQ、TFO 等）。

整体上，这些结构体现了分层与复用：同一层内用“第一个成员是父类”做 C 风格继承，跨层用指针关联（skb↔dev、skb↔sk），既避免重复实现，又保持协议栈清晰、可维护和可扩展。
