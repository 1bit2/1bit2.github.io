一、对齐的三个维度

内核中的对齐不仅仅是"让访问快一点"，它服务于三个完全不同的目标：

维度核心诉求不满足的后果
正确性DMA buffer 满足 cache line 边界；原子指令满足操作数对齐数据损坏、CPU 异常
性能热数据独占 cache line；消除 false sharing多核场景性能下降 10x
可移植性满足架构最小对齐约束（如 ARM 不支持未对齐访问）运行时 fault

澄清：不对齐导致的是"多访问一条 cache line"，不是 cache 失效

一个常见误解是"字节不对齐会导致 cache 失效"。错。 Cache 失效（Invalidate）只由 MESI 协议的一致性消息、容量驱逐、或软件显式刷新触发，与对齐无关。

不对齐的真正代价是跨 cache line 访问：

对齐访问（8 字节 long 在地址 0x40）：
┌──────────────── Cache Line 1 (0x40~0x7F) ────────────────┐
│  ... │ [long 在这，完整落在一条 line 内] │ ...             │
└──────────────────────────────────────────────────────────┘
  → 一次 cache line 访问

不对齐访问（8 字节 long 在地址 0x3C，跨越 0x3F/0x40 边界）：
┌────── Cache Line 0 (0x00~0x3F) ──────┐┌────── Cache Line 1 (0x40~0x7F) ───────┐
│                       │ [前4字节]      ││ [后4字节] │                            │
└────────────────────────────────────────┘└───────────────────────────────────────┘
  → 两次 cache line 访问 + CPU 内部拼接
场景代价
跨 line 访问（x86）延迟翻倍（如果第二条 line miss 则更糟）；占用两个 cache line 空间
跨 line 访问（ARM/MIPS）部分架构直接触发对齐异常，由软件模拟（极慢）或 fault

Cache 失效（Invalidate）的真正触发条件：

触发条件机制
其他核写了同一物理地址MESI: Shared → 收到 Invalidate → Invalid
容量/冲突驱逐LRU 淘汰旧 line
软件显式刷新clflush/wbinvd/invd 指令
DMA 一致性操作dma_sync_for_cpu 等

  __attribute__((__aligned__(SMP_CACHE_BYTES), \
                 __section__(".data..cacheline_aligned")))

设计细节：

  • ____cacheline_aligned_in_smp：SMP 内核下按 cache line 对齐，单处理器内核下为空（因为没有 false sharing 问题）
  • __cacheline_aligned：额外放入专用段 .data..cacheline_aligned，便于链接器统一布局

// mm/slab_common.c:146
static unsigned int calculate_alignment(slab_flags_t flags,
        unsigned int align, unsigned int size)
{
    if (flags & SLAB_HWCACHE_ALIGN) {
        unsigned int ralign = cache_line_size();
        // 小对象减半对齐：16B 对象不需要占满整条 64B cache line
        while (size <= ralign / 2)
            ralign /= 2;
        align = max(align, ralign);
    }

    if (align < ARCH_SLAB_MINALIGN)
        align = ARCH_SLAB_MINALIGN;

    return ALIGN(align, sizeof(void *));
}

规则链：

  1. 用户传入 align 为下限
  2. SLAB_HWCACHE_ALIGN 提升到 cache line 大小（小对象减半）
  3. 不低于 ARCH_SLAB_MINALIGN
  4. 最终对齐到指针大小

四、DMA 对齐：不是性能问题，是正确性问题

问题本质

设备 DMA 直接写物理内存，不经过 CPU cache。如果 DMA buffer 与其他数据共享一条 cache line：

Cache Line [64 bytes]
┌──────────────────────────────────────┬──────────────────┐
│  DMA buffer (设备正在写入)            │  其他变量 X      │
└──────────────────────────────────────┴──────────────────┘
                                              ↑
                                     CPU 读 X，把整条 line 拉入 cache
                                     设备写完 DMA buffer
                                     CPU 写回 cache line → 覆盖 DMA 数据！

六、False Sharing 防护实战

视角分工：本节只讲内核侧的 false sharing 防护实例（XDP 环形队列、TCP TIME_WAIT 计数器、BPF map 引用计数、HTB 流控、网络哈希表锁）。

关于 MESI 状态机层面的原理、用户态矩阵遍历案例、perf c2c 定位方法等软件视角内容，详见 CPU 页表与缓存机制 §19。

False sharing：不同 CPU 修改同一 cache line 内的不同变量，导致 cache line 在核间反复失效，实际独立的操作变成了串行。

为什么有了自然对齐还需要手动写对齐？

编译器的自然对齐（N 字节类型对齐到 N 的倍数）解决的是单个变量不跨 cache line；手动对齐解决的是不同核写的变量不在同一条 cache line。两者目标完全不同：

自然对齐（编译器自动保证）:
  ┌──────────── 64 字节 Cache Line ─────────────┐
  │ [int_a] [int_b] [int_c] ... [int_d]         │
  └──────────────────────────────────────────────┘
  Core 0 写 int_a → 整条 line Invalidate → Core 1 读 int_d 被波及
  
  ✓ 单个 int 不会跨 line（自然对齐的功劳）
  ✗ 但多个变量挤在同一 line 内（false sharing）

手动 cache line 对齐（程序员显式声明）:
  ┌── Cache Line 0 ──┐  ┌── Cache Line 1 ──┐
  │ int_a + 60B pad   │  │ int_d + 60B pad   │
  └───────────────────┘  └───────────────────┘
       Core 0 独占             Core 1 独占
       
  ✓ 各核的写操作互不触发 Invalidate

总结：

自然对齐（编译器自动）手动 cache line 对齐
对齐到sizeof(type) 字节（int→4, long→8）64 字节（整条 cache line）
保证一次访问不跨两条 line不同核的写不在同一条 line
代价极小（结构体内部 padding 几字节）较大（每个字段浪费最多 60 字节）
自动 vs 手动编译器自动完成程序员显式声明 ____cacheline_aligned_in_smp
解决的问题类别正确性+单核性能多核并发性能

关键洞察：4 字节的 u32 自然对齐到 4 的倍数——它永远不可能"跨" 64 字节边界（4 < 64），但它完全可能和另一个高频写变量共处一条 cache line。手动对齐解决的是后者。

“对齐"在 false sharing 语境下的含义

内核中"对齐"用于两个不同目的，容易混淆：

目的手段解决的问题
自然对齐变量起始地址对齐到 sizeof(type) 边界避免一次访问跨两条 cache line（多一次加载）
独占对齐变量占满/对齐到整条 cache line（64B）避免与其他核的写变量共享 line（MESI Invalidate）

____cacheline_aligned_in_smp 做的是后者——它不是怕变量"跨 line”（4 字节 u32 不可能跨 64B 边界），而是要让这个变量独占一整条 line，使不同核的写操作落在不同 line 上。

假设 producer 和 consumer 没有对齐（同在一条 cache line）：

┌────────────── 同一条 Cache Line ──────────────┐
│  producer (Core 0 写)  │  consumer (Core 1 写) │
└───────────────────────────────────────────────┘

t1: Core 0 写 producer
    → 该 line 在 Core 0 上变为 Modified (M)
    → MESI 发 Invalidate 给 Core 1
    → Core 1 的这条 line 变为 Invalid (I)

t2: Core 1 要写 consumer
    → 发现自己的 line 是 Invalid → 必须从 Core 0 取最新数据
    → Core 0 写回（M→I），Core 1 获得 line（I→M）
    → Core 1 写 consumer
    → 发 Invalidate 给 Core 0 → Core 0 的 line 变 Invalid

t3: Core 0 要写 producer
    → 又是 Invalid → 又要从 Core 1 拿回来...

→ 每次写操作都要等 30-100ns 的跨核传输（cache-to-cache transfer）
→ 本来可以各自 1ns 完成的写，变成串行的 30-100ns

对齐后各自独占一条 line：

┌──── Cache Line A ────┐  ┌──── Cache Line B ────┐
│ producer + 60B pad    │  │ consumer + 60B pad    │
└───────────────────────┘  └───────────────────────┘
     Core 0: Modified            Core 1: Modified
     
互不干扰：Core 0 写 producer 不触发任何 Invalidate
         Core 1 写 consumer 不触发任何 Invalidate

struct xdp_ring {
    u32 producer ____cacheline_aligned_in_smp;
    /* Hinder the adjacent cache prefetcher to prefetch the consumer
     * pointer if the producer pointer is touched and vice versa. */
    u32 pad1 ____cacheline_aligned_in_smp;

4 字节的 producer 独占 64 字节 cache line。为什么？

  • 内核写 producer，用户空间写 consumer
  • 若共享 cache line：每次一端写入，另一端的 CPU cache 失效 → 重新加载
  • 高频路径（数百万包/秒）下，false sharing 可导致吞吐量减半

七、对齐的空间代价与权衡设计

对齐是有代价的——padding 浪费空间。内核做了精细的权衡：

1. 小对象减半对齐

calculate_alignment() 中：

1
2

while (size <= ralign / 2)
    ralign /= 2;

16 字节对象不需要 64 字节对齐（浪费 75%），对齐到 16 字节即可——一条 cache line 放 4 个对象，命中率不差，空间利用率高。

2. UP 下消除 SMP padding

____cacheline_aligned_in_smp 在单处理器内核下为空：没有多核就没有 false sharing，不需要浪费空间。

八、设计思考

1. 为什么 ARM64 的 ARCH_DMA_MINALIGN 是 128 而不是 64？

ARM64 的 Cache Writeback Granule（CWG）可以大于 L1 cache line。CWG 是 CPU 单次写回内存的最小粒度——如果 DMA buffer 不满足 CWG 对齐，CPU 写回相邻数据时可能覆盖 DMA 正在写入的区域。128 是 ARM big.LITTLE 架构中常见的最大 CWG。

2. 为什么 x86 不需要那么大的 DMA 对齐？

x86 有 cache coherent DMA（设备通过 snoop 协议参与 cache 一致性），不存在 CPU 回写覆盖 DMA 数据的问题。因此 ARCH_KMALLOC_MINALIGN 只需要 8 字节（unsigned long long 的自然对齐）。

3. 对齐的三层递进

层1：最小正确性对齐（ARCH_SLAB_MINALIGN = 8）
  → 保证基本类型访问不触发异常

层2：DMA 安全对齐（ARCH_KMALLOC_MINALIGN = 128 on ARM64）
  → 保证 kmalloc buffer 可安全用于 DMA

层3：性能优化对齐（SLAB_HWCACHE_ALIGN / ____cacheline_aligned_in_smp）
  → 消除 false sharing，提升多核并发性能
